## What the repo is for

It is not meant to be a magic prompt library. It is more like a working notebook
for how I want a coding agent to behave: how much context it should read before
changing code, when it should plan, what kind of diffs I want back, and which
repeated workflows are worth turning into commands or agents.

The main `CLAUDE.md` follows the [Karpathy-style principles](https://x.com/karpathy/status/2015883857489522876) I keep coming back to:
make the agent read context first, keep the solution simple, change only what is
needed, and verify the result with evidence instead of confidence.

<div className="note-callout">
  <strong>The point:</strong> make the boring parts repeatable so I can spend
  more attention on the actual decision-making.
</div>

The exact files will keep changing. That is the point of having it in Git. I can
try things, keep what survives real work, and delete what only sounded clever in
the abstract.

## My working loop

<ol className="note-steps">
  <li>
    <span>01</span>
    <div>
      <strong>Read first.</strong>
      Ask the agent to inspect the relevant files, callers, tests, and config
      before it suggests a change.
    </div>
  </li>
  <li>
    <span>02</span>
    <div>
      <strong>Plan when useful.</strong>
      For non-trivial work, I want a short plan and a verification step before
      code changes start.
    </div>
  </li>
  <li>
    <span>03</span>
    <div>
      <strong>Keep diffs small.</strong>
      The bigger the agent-generated diff, the harder it is to trust.
    </div>
  </li>
  <li>
    <span>04</span>
    <div>
      <strong>Verify with evidence.</strong>
      Tests, builds, screenshots, or clear manual checks beat confidence.
    </div>
  </li>
</ol>

## A prompt pattern I reuse

<div className="note-code">
  <div className="note-code-head">
    <span>prompt pattern</span>
    <code>plan-first</code>
  </div>
  <pre><code>{`Read the relevant files before editing.

Then give me:
1. What you found
2. The smallest safe plan
3. The verification command or manual check

Do not change code until the plan is clear.`}</code></pre>
</div>

That pattern sounds basic, but it prevents a lot of bad sessions. The agent is
most useful when it is anchored to the repo instead of guessing from a vague
request.

## What I put in commands

Commands are useful when a prompt becomes a habit. If I type the same instruction
three or four times, it probably belongs in the setup.

<div className="note-code">
  <div className="note-code-head">
    <span>command idea</span>
    <code>/review-diff</code>
  </div>
  <pre><code>{`Review the current diff.
Focus on:
- bugs or broken assumptions
- unnecessary complexity
- missing verification
- changes outside the requested scope

Return only actionable notes.`}</code></pre>
</div>

## The habits that matter most

<div className="note-chip-row">
  <span>Read before editing</span>
  <span>Small diffs</span>
  <span>Explicit verification</span>
  <span>Review everything</span>
</div>

The biggest improvements have not come from one perfect prompt. They have come
from boring constraints repeated consistently. Claude Code is fast, but speed is
not the same as judgment. The setup is there to make it a better collaborator,
not to remove taste, review, or responsibility.
